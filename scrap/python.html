using chain of decor, write code for login start, user auth, money transf, log ended?

def logging(func):
    def wrapper():
        print("logging started")
        func()
        print("logging ended")
    return wrapper

def authentication(func):
    def wrapper():
        print("user authenticated")
        func()
    return wrapper

@logging
@authentication
def transfer_money():
    print("money transferred")

app = Flask(__name__)

students = []

@app.route("/", methods=["GET"])
def get_students():
    return jsonify(students)

@app.route("/", methods=["POST"])
def create_student():
    data = request.get_json()

    if len(data["name"]) < 3:
        return jsonify({
            "error": "Name must be at least 3 characters"
        }), 400

    students.append(data)

    return jsonify({
        "message": "Student saved"
    }), 201

if __name__ == "__main__":
    app.run(debug=True)

--gRPC

gRPC = high-performance RPC framework using HTTP/2 and Protocol Buffers
Why is protobuf preferred over json = smaller, faster to serialize/deserialize, and gives a strict schema (performance up)
retries how works = handle transient failures using backoff, max attempts, and jitter
idempotency = runs multiple times, same effect (no dup work)
-- scenario quetions
event received twice due to retry - avoid duplicates = Use an idempotency key, store processed event IDs
A gRPC service is fast, but the automation is slow. Where do you look first? = check downstream events, queue backlog
single bad event keeps failing = Move it to a dead-letter queue
handle schema changes in protobuf? = Use backward-compatible changes
workflow has steps A -> B -> C. B fails after A succeeds. What pattern helps? = Saga pattern or compensating actions. Persist step state so the workflow can resume safely
gRPC vs REST = Think HTTP/2 + Protobuf + streaming + high performance (gRPC) vs HTTP/JSON + browser-friendly + wider interoperability (REST).

producer = apps that publish messages to kafka - creae producer record
what zookeeper does = used for distributed coord and mgt of kafka cluster (maintains metadata, leader election)
ISR Replicas = In Sync Replicas - that maintains the replica 
(practical) - 6 partitions - how many consumer instnces recommended = 6 (1 each)
(practical) - if only 2 consumer instances = C1: {p0, p1, p2}, c2: {p3, p4, p5}; supose 4 consumers? - may be C1: {p0, p1}, C2: {p2, p3}

(practical) - if one consumer dies = kafka reconfigures and rebalancing
how many copy of events (E.g. Hello world) = depends on replication factor (E.g. 3) - Leader copy=1, Follower copy = 2
if orderCreated, orderShipped both go to different partitions, wont consumer c1 picks orderCreated after c2 picking orderShipped = yes, may be (kafka no guarantee) - so, devs use Order_ID=101 (define key) for all events and put one by one in same partition
